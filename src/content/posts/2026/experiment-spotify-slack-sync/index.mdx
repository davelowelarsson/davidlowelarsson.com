fredagslistan started as a Slack ritual: every Friday, someone sets a theme in the morning, people drop songs that fit it, and the playlist grows.

"Fit" is doing generous work there. Free interpretation is not a loophole. It is basically the rule.

A few anonymized and translated openings, stripped of Slack links and names, give the shape better than architecture prose can:

> There is a match today, so the theme is **your best match/training hype songs**! ⚽ 🥵 🤸‍♀️ Free interpretation!

> Friday! Today's theme is **flashbacks from lower secondary school**. Time for a mix of teenage personalities and decades. Let's go!

> Surely nobody missed the music-making freezers in England? Apparently fantastic music in C#. So today the theme is **instrumental music**! 🥁 🎸

> Toot toooot! Today's Fredagslistan theme is **trains**. Go go go.

For a while the playlist was maintained by hand. That worked about as well as hand-maintained shared state usually works: fine when someone remembered, wrong the moment they did not.

Most people dropped tracks where the conversation already was, in Slack, and never made the second hop into Spotify. So the playlist was always a little wrong.

I wrote a small Python cronjob to do the transfer for us. It has been running for years now. Quietly. No heroics. Exactly what I wanted.

At some point it escaped work, too. At Friday dinners with friends I started asking them to guess the theme from the playlist. A normal person might call that small talk. I apparently turned it into an integration problem.

## One run

### 1. Fetch from Slack

The job reads the Friday channel, extracts track links, and normalizes IDs. This is mostly input hygiene, because chat data is messy in ways APIs are not.

<ProcessStepCard step={syncSteps.fetchSlack} />

### 2. Compute the diff

Next step is simple set math: what exists in Slack but not yet in Spotify. No scoring model, no ranking, no "smart" guesswork. Just the gap.

<ProcessStepCard step={syncSteps.diff} />

### 3. Add missing songs

Missing tracks are appended to the playlist. If a song is already there, the run leaves it alone. Re-runs are intentionally boring.

<ProcessStepCard step={syncSteps.addSpotify} />

There is one small design choice here that I like more than it deserves: the job only looks back a few days in Slack.

That six-day rolling window is enough to catch the current Friday and late additions without turning the whole channel history into input every time. It also means an old Spotify link someone reposts weeks later is treated like a new suggestion, which is probably correct. The source of truth is not "all links ever seen." It is "what people are talking about around this week's theme."

## Snapshot, not surveillance

For this write-up I only want aggregate signals. No names. No leaderboard. No accidental coworker analytics.

<WrappedSnapshot stats={wrappedStats} />

The honest footnote is that some of the operational stats I now want are not historical, because I did not log every run from the start. If I want to say "auto-added over the years" later, I need the job to write that number down from here on instead of pretending a current playlist snapshot is the same thing as a run log.

That is fine. A small tracks-per-year curve, total listening time, returning artists, or genre drift would probably be more interesting than a fake precision counter anyway.

That is probably the real reason I still like this little job. It is not clever infrastructure. It is a tiny bit of glue that protects a ritual from becoming admin work.
